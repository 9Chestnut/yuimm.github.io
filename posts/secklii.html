秒杀防止库存超卖(redis)

第一种：使用synchronized

    @Autowired
    StringRedisTemplate redisTemplate; 

    public String buy(){
            synchronized (this) {
                int num = Integer.parseInt(redisTemplate.opsForValue().get("phone"));
                if (num > 0) {
                    int newNum = num - 1;
                    // 更新库存-redis
                    redisTemplate.opsForValue().set("phone", String.valueOf(newNum));
                    System.out.println("扣减库存成功，剩余库存为：" + newNum);
                } else {
                    System.out.println("扣减失败");
                }
            }
        return "end";
    }

synchronized锁是基于jvm的，只能针对单个jvm进行上锁，当出现服务器集群（多个服务器）的时候【使用Nginx负载两台服务器的时候】，不能保证并发安全。

第二种：使用redis分段锁

版本1：

  public String buy(){
        String lockKey = "LockKey";
        try {
            // setIfAbsent() 类似于redis的setnx，只有当key不存在的时候才能创建，key存在的时候无法创建
           Boolean result = redisTemplate.opsForValue().setIfAbsent(lockKey, "hct");
           redisTemplate.expire(lockKey,10,TimeUnit.SECONDS); 
            if (!result){
                return "error";
            }
            int num = Integer.parseInt(redisTemplate.opsForValue().get("phone"));
            if (num > 0) {
                int newNum = num -1;
                // 更新库存-redis
                redisTemplate.opsForValue().set("phone",String.valueOf(newNum));
                System.out.println("扣减库存成功，剩余库存为：" + newNum);
            } else {
                System.out.println("扣减失败");
            }
        } finally {
            redisTemplate.delete(lockKey);
        }

        return "end";
    }
  1. 使用 try-finally 是为了防止其中一个线程还没执行到第20行代码的时候抛出了异常，导致 lockKey 无法删除，而后面的线程会一直卡在地5行代码导致业务无法继续进行。

版本2：

  1. 在使用了try-finally解决了 1 的问题之后，如果其中一个线程还未执行完，其中一台正在运行的服务器突然断电或者宕机了，而finally也永远无法执行，后续的线程被负载均衡到其他的服务器也会被卡在第5行代码无法执行，所以要把第5行和第6行代码合为一条代码，要么都成功，要么不成功：Boolean result = redisTemplate.opsForValue().setIfAbsent(lockKey, "hct",10,TimeUnit.SECONDS);

此时可能还会出现的问题：其中一个线程业务还没执行完，锁就被释放了,导致线程去释放其他线程的锁，导致锁失效！！！

假设业务执行需要 15 秒，第一个线程进来，当执行到第 9 行代码的时候已经花费了10秒，此时线程1加的锁就失效了，此时线程2进来了，且可以成功加锁，线程1继续执行5秒后会执行 finally 中的第 19 行代码，此时线程1释放的却是线程2加的锁，以此类推线程3，线程4……导致了琐失效。

版本3：

解决办法

使用 clientId（每个进来秒杀的用户的id进行判断），此时每个线程都只能释放自己的锁，无法释放别人的锁。

此时可能还会出现的问题：锁失效的时间问题，其中一个线程业务还没执行完，锁还是被释放了，其他线程又可以顺利加锁！！！

    public String buy(){
        String lockKey = "LockKey";
        String cliendId = UUID.randomUUID().toString();
        try {

            Boolean result = redisTemplate.opsForValue().setIfAbsent(lockKey, cliendId,10,TimeUnit.SECONDS);
            if (!result){
                return "error";
            }
            int num = Integer.parseInt(redisTemplate.opsForValue().get("phone"));
            if (num > 0) {
                int newNum = num -1;
                // 更新库存-redis
                redisTemplate.opsForValue().set("phone",String.valueOf(newNum));
                System.out.println("扣减库存成功，剩余库存为：" + newNum);
            } else {
                System.out.println("扣减失败");
            }
        } finally {
            if (cliendId.equals(redisTemplate.opsForValue().get(lockKey))){
                redisTemplate.delete(lockKey);
            }
        }
        return "end";
    }

解决办法

开辟一个子线程去统计主线程的业务有没有执行完，如果没有执行完，就不断的累加锁失效的时间（续命的方式）【Redisson给提供了解决方案】

导入redis的依赖

        <dependency>
            <groupId>org.redisson</groupId>
            <artifactId>redisson</artifactId>
            <version>3.13.6.5</version>
        </dependency>

设置bean

     @Bean
    public Redisson redisson(){
        Config config = new Config();
        config.useSingleServer().setAddress("redis://192.168.36.128:6379").setDatabase(0);
        return (Redisson) Redisson.create(config);
    }

可能出现的问题：Redis集群

当一个redis的时候，redisson是没有问题的，但是当出现redis通过主从复制的搭建的集群时，当一个线程为redis主机进行加锁完毕之后，主机还没来得及向从机同步锁就宕机了，此时从机中是没有锁的，所以当另一个线程去访问从机的时候又可以成功加锁！！！

Redis缓存与数据量双写不一致

一：读多写少

使用读写锁可以大大的提高效率，读锁不会影响，写锁才会串行化执行。
